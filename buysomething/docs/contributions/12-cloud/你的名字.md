### 3. 问题解决

- 遇到的问题：
  1. CloudBase 环境需要手动创建，初始配置较复杂
  2. GitHub Actions 需要配置 API 密钥才能访问 CloudBase
  3. Docker 镜像需要配置正确的 HEALTHCHECK 确保容器健康
  4. 多平台部署需要统一的配置文件管理

- 解决方案：
  1. 通过 CloudBase MCP 工具自动化环境查询，避免手动配置
  2. 在 GitHub Secrets 中安全存储 API 密钥，工作流通过 `${{ secrets.XXX }}` 引用
  3. 在 Dockerfile 中添加 `HEALTHCHECK` 指令，配合 `/health` 端点实现容器健康自检
  4. 使用 `vercel.json` + `Dockerfile` + GitHub Actions 三层配置覆盖所有部署场景

## 心得体会

1. **Serverless 架构优势**：CloudBase 云函数免去了服务器运维的负担，配合 GitHub Actions 实现全自动 CI/CD，代码推送即部署，极大提升开发效率。

2. **多平台部署的价值**：同时配置 Vercel 和 Docker/GHCR 两种部署方式，前者适合快速预览和演示，后者适合生产环境的容器编排，灵活性很强。

3. **安全敏感信息管理**：通过 GitHub Secrets 和环境变量管理 API 密钥、数据库连接串等敏感信息，避免硬编码带来的安全风险，是生产环境部署的基本要求。

4. **CI/CD 自动化的重要性**：代码检查（ESLint）→ 安全扫描（Gitleaks）→ 构建部署，完整的流水线确保每次提交都经过质量把关，减少线上故障。

5. **Docker 多阶段构建**：利用多阶段构建将最终镜像从数百 MB 压缩到 ~80MB，显著减少拉取和启动时间，降低资源消耗。
