===========================================
作业二：监控配置 — 遇到的问题和解决
===========================================

【成员 A 使用 1-4 条，成员 B 使用 5-8 条】


--- 遇到的问题和解决 ---

1. 问题：健康检查端点实现到什么程度才算"真正的健康检查"，而不只是一个返回固定值的假端点
   解决：参考业界最佳实践，健康检查需要验证关键依赖的可用性。在 health 云函数中增加了数据库连接检测——调用 db.collection('products').count() 真实查询数据库，如果数据库不可用则返回 degraded 状态（HTTP 503），只有所有检查通过才返回 healthy（HTTP 200）。

2. 问题：云函数是无状态环境，每次冷启动内存都会清空，如何实现有意义的指标统计
   解决：设计了 MetricsCollector 类进行当次实例级别的内存统计，通过 getSummary() 方法实时计算当前调用的指标（请求数、响应时间、错误率）。同时文档中说明了对于长期统计（跨实例），应将指标数据写入 CloudBase 数据库做持久化存储，或对接外部监控平台。

3. 问题：结构化日志的 JSON 格式可能在 CloudBase 日志中心无法被正确解析和检索
   解决：确保日志输出的 JSON 对象是一行一行的纯 JSON（不含换行），使用 console.log(JSON.stringify(entry)) 而非 console.log(entry)，这样 CloudBase 的日志采集系统能将其识别为结构化数据。同时为每条日志增加了 level 和 module 固定字段，方便按日志级别和模块名称过滤。

4. 问题：监控工具代码（monitor-utils）如何在多个云函数之间共享
   解决：CloudBase 每个云函数是独立部署的代码包，无法像传统 Node.js 项目那样 import 共享模块。解决方案有两种：一是将工具代码复制到每个需要的云函数目录中，二是将工具库发布为 CloudBase Layer（层），让多个函数绑定同一层。文档中说明了两种方案的适用场景。

5. 问题：健康检查 cloud 函数应该部署为 Event 类型还是 HTTP 类型，初次部署时报错
   解决：最初尝试部署为 HTTP 类型时报 "ResourceNotFound.BootstrapFile" 错误，因为 HTTP 型云函数需要额外的运行时引导文件。改为 Event 类型部署成功，同时函数内部通过判断 event.httpMethod 来兼容 HTTP 调用和小程序 callFunction 两种方式，灵活性更好。

6. 问题：日志中可能无意中泄露用户敏感信息（手机号、密码、Token 等）
   解决：在 StructuredLogger 中内置了数据脱敏逻辑——自动将手机号中间四位替换为 ****，自动过滤 password 字段为 ***。所有日志输出前经过脱敏处理，确保即使开发者不小心传入了敏感数据，日志中也不会泄露。

7. 问题：前端 Express 服务的健康检查和云函数的健康检查如何统一输出格式
   解决：制定了一个统一的健康检查响应规范：{ status, timestamp, version } 为必填字段，checks 为可选的检查项详情。前端 server.js 的 /health 和云函数 health 的返回值都遵循这个规范，方便监控系统统一解析。

8. 问题：docker-compose 中配置的 HEALTHCHECK 指令从未验证过是否真的生效
   解决：通过 docker inspect 命令查看了容器的健康状态，确认 Docker 定期调用 /health 端点检查，当 Express 服务正常时返回 healthy，服务挂掉时容器状态变为 unhealthy 并触发重启策略。这个验证确保了容器编排层面的健康检查是真实可用的。


--- 心得体会 ---

1. 可观测性是运维的基础：写代码只是第一步，服务上线后"能不能看到它在干什么、有没有出问题"同样重要。结构化日志、健康检查、指标收集这三样构成了服务可观测性的骨架，缺了任何一环都会在出故障时变成"瞎子摸象"。

2. 日志格式标准化不是小题大做：相比 console.log('xxx failed') 这种随意写法，JSON 格式的日志可以在 CloudBase 控制台按 level:ERROR 精确过滤、按 module:login 定位模块、按时间范围聚合分析。日志从"给人看的"变成"给机器查的"，效率天差地别。

3. Serverless 监控的特殊挑战：传统服务器上的监控工具（如 Prometheus exporter）在云函数环境中无法直接使用，因为函数实例是临时的、无状态的。这促使我思考如何在"短暂生命周期"中完成有效的指标采集——答案是在函数内部做轻量级的内存统计，在函数返回前输出汇总。

4. 健康检查不能"骗自己"：最初的想法是直接返回 { status: 'ok' }，但这只能证明"函数代码在跑"，不能证明"数据库能连"、"API 能响应"。加入了数据库连接检测后，健康检查才真正反映了服务的真实状态，这样的监控才有意义。

5. 监控代码也是业务代码的一部分：以前总觉得监控是运维的事、是额外的工作量。实际上，把日志埋点和指标收集写在业务代码旁边，出了问题能第一时间定位，这本身就是开发效率的提升——花 10% 的时间写监控，省 90% 的排查时间。

6. 云平台的原生能力要善用：CloudBase 自带了日志采集、检索、函数监控面板，不需要自己搭建 ELK 或 Grafana。在了解平台已有能力的基础上做增强（如统一日志格式、增加自定义指标），而不是从头造轮子，是最务实的做法。

7. Docker 健康检查的巧妙之处：Dockerfile 里的 HEALTHCHECK 指令配合 /health 端点，实现了容器级别的自愈——服务挂了 Docker 会自动重启，不需要人工干预。这让我意识到监控的真正价值不在于"发现问题"，而在于"自动响应"。

8. 敏感信息脱敏要自动化：靠开发者的自觉（"记着别打印密码"）是不可靠的。在日志工具层做自动脱敏——手机号中间位遮挡、密码字段替换——才能保证任何时候都不会出事。这是从"人治"到"机制"的转变。

9. 指标收集的层次设计：请求计数是最外层（每次调用 +1），响应时间是在函数体执行前后打时间戳，错误率是有没有进 catch 块。这些指标不需要复杂的统计库，几个变量 + 一点计算就能实现，关键在于"有"而不在于"全"。

10. 监控和部署是一体两面：部署让服务上线，监控让服务稳定。这次作业 1（云部署）和作业 2（监控配置）前后衔接得很好——部署完成后立刻配置监控，就像买了房子立刻装门锁和监控摄像头，是顺理成章的工程实践。
