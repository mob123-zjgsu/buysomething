## 遇到的问题和解决

### 问题 1：云函数依赖缺失

**问题描述**：
```
Cannot find module 'wx-server-sdk'
云函数调用失败，错误码 -504002
```

**问题分析**：
测试时发现所有云函数都无法正常调用，控制台显示找不到 `wx-server-sdk` 模块。这是一个依赖配置问题，导致云函数无法正常启动。

**排查过程**：
1. 检查云函数代码，发现使用的是 `require('tcb-admin-node')`
2. 检查 `package.json`，发现依赖配置为 `"wx-server-sdk": "^2.6.3"`
3. 代码引用的模块名与依赖配置不一致

**解决过程**：
1. 将 `package.json` 中的依赖改为 `tcb-admin-node`
2. 重新部署云函数
3. 测试发现仍然报错
4. 再次排查，发现代码需要改为 `wx-server-sdk` 才能正常工作
5. 最终决定：代码改用 `wx-server-sdk`，保持 `package.json` 一致

**解决代码**：
```javascript
// 修改前（错误）
const tcb = require('tcb-admin-node');

// 修改后（正确）
const cloud = require('wx-server-sdk')
cloud.init({ env: 'buysomething-6gbmbtpxff05be35' })
```

**经验总结**：
微信小程序云函数应该使用 `wx-server-sdk`，这是官方提供的 SDK。而 `tcb-admin-node` 适用于其他场景。

### 问题 2：登录密码验证失败

**问题描述**：
使用正确的密码 `123456` 登录，测试显示"密码错误"

**问题分析**：
登录功能测试时，无论输入什么密码都返回"密码错误"，但密码确实是正确的。这说明验证逻辑可能存在问题。

**排查过程**：
1. 检查登录云函数代码
2. 发现密码比对逻辑：`user.password !== password`
3. 检查数据库中的用户数据
4. 发现数据库中存储的密码是：`e10adc3949ba59abbe56e057f20f883e`
5. 这是 MD5 加密后的密码，但代码直接比对明文

**解决过程**：
1. 添加 MD5 加密函数
2. 将用户输入的密码转为 MD5 后再比对
3. 重新部署云函数
4. 测试登录成功

**解决代码**：
```javascript
const crypto = require('crypto')

// MD5 加密函数
function md5(str) {
  return crypto.createHash('md5').update(str).digest('hex');
}

// 验证密码
const inputPasswordMd5 = md5(password);
if (user.password !== inputPasswordMd5) {
  return { code: 1002, message: '密码错误' }
}
```

**经验总结**：
数据库存储密码时通常会加密（如 MD5、SHA256），验证时需要将用户输入的密码进行相同加密后再比对。这是常见的安全做法。

### 问题 3：商品不存在时返回服务器错误

**问题描述**：
测试"商品不存在"场景时，期望返回 `code: 1004`（商品不存在），但实际返回了"服务器错误"

**问题分析**：
这个错误发生在使用 `doc()` 方法查询商品时。当传入的 `productId` 格式不正确时，会导致数据库查询抛出异常，而不是返回"不存在"的结果。

**排查过程**：
1. 测试时使用的商品 ID：`non-existent-id-12345`
2. 这个 ID 不是有效的 MongoDB ObjectId 格式
3. 使用 `db.collection('products').doc(productId).get()` 查询时
4. CloudBase 数据库认为无效 ID 格式会导致查询失败
5. 异常被 catch 捕获，返回了通用的"服务器错误"

**解决过程**：
1. 改用 `where({ _id: productId })` 查询方式
2. 这种方式对 ID 格式更宽容
3. 查询不到时返回空数组，而不是抛出异常
4. 检查数组长度来判断商品是否存在
5. 重新部署后测试通过

**解决代码**：
```javascript
// 修改前（会报错）
const queryResult = await db.collection('products').doc(productId).get();
if (queryResult.data.length === 0) {
  return { code: 1004, message: '商品不存在' }
}

// 修改后（更健壮）
const queryResult = await db.collection('products').where({
  _id: productId
}).get();

const productList = queryResult.data || [];
if (productList.length === 0) {
  return { code: 1004, message: '商品不存在' }
}
```

**经验总结**：
使用数据库查询时，应该选择更健壮的查询方式。当查询条件可能无效时，应该让代码自然返回空结果，而不是抛出异常。

### 问题 4：WXML 模板语法错误

**问题描述**：
```
Bad value with message: unexpected token `.`
Error: webviewScriptError
__route__ is not defined
```

**问题分析**：
在 WXML 模板中使用了 JavaScript 方法调用 `.toFixed()`，但 WXML 模板不支持这种语法。

**排查过程**：
1. 检查报错的 WXML 文件
2. 发现代码：`<text>{{(passedTests / totalTests * 100).toFixed(0)}}%</text>`
3. WXML 是微信小程序的模板语言，不是完整的 JavaScript
4. WXML 只支持简单的三元运算符和基本的属性访问
5. 不支持调用 JavaScript 对象的方法

**解决过程**：
1. 在 JS 的 `data` 中添加 `passRate` 字段
2. 在 `addResult` 函数中计算通过率并更新
3. WXML 中直接使用变量：`{{passRate}}`
4. 重新编译，问题解决

**解决代码**：
```javascript
// JS 中计算
data: {
  passRate: 0,
  // ...
}

// 计算通过率
const passRate = totalTests > 0 
  ? Math.round((passedTests / totalTests) * 100) 
  : 0

this.setData({ passRate })

// WXML 中使用
<text>{{passRate}}%</text>
```

**经验总结**：
微信小程序的 WXML 模板语法有限制，不能直接调用 JavaScript 方法。所有计算逻辑都应该在 JS 中完成，然后通过 `setData` 更新到视图层。

### 问题 5：测试数组未定义

**问题描述**：
```
Cannot read property 'push' of undefined
测试无法正常记录结果
```

**问题分析**：
在 `addResult` 函数中尝试向 `this.data.apiResults` 数组添加元素时，该数组尚未初始化。

**排查过程**：
1. 检查 `addResult` 函数
2. 发现代码：`this.data.apiResults.push(result)`
3. 检查 `data` 的初始化
4. 发现 `initTestPage` 中使用 `setData` 初始化了数组
5. 问题出在 `onLoad` 时调用了测试函数，但数组可能还未初始化完成

**解决过程**：
1. 在 `addResult` 中添加数组存在性检查
2. 如果数组不存在，初始化为空数组
3. 确保任何时候都能正确添加结果

**解决代码**：
```javascript
addResult(category, name, passed, message, duration = 0) {
  const key = category + 'Results'
  
  // 确保数组存在
  let results = this.data[key]
  if (!results || !Array.isArray(results)) {
    results = []
  }
  results.push(result)
  
  this.setData({ [key]: results })
}
```

**经验总结**：
在小程序中，`setData` 是异步操作，可能存在时序问题。访问数据前应该检查其是否存在，必要时进行初始化。

### 问题 6：创建订单失败

**问题描述**：
测试创建订单时，期望成功返回订单 ID，但实际返回"商品信息无效"

**问题分析**：
订单创建需要商品信息来计算总价，如果商品 ID 无效或不存在，订单创建会失败。

**排查过程**：
1. 检查订单创建的测试代码
2. 发现使用的是 `this._testProductId`
3. 追踪变量来源
4. 发现商品 ID 是从"获取商品列表"测试中保存的
5. 可能存在测试执行顺序问题，或商品列表为空

**解决过程**：
1. 在创建订单测试中先检查是否有商品 ID
2. 如果没有，先调用商品列表 API 获取
3. 获取成功后再创建订单
4. 添加详细的日志输出，方便排查

**解决代码**：
```javascript
async testAPI_CreateOrder() {
  // 确保有可用的商品ID
  let productId = this._testProductId
  if (!productId) {
    // 先获取商品列表
    const productsRes = await wx.cloud.callFunction({
      name: 'products',
      data: { page: 1, pageSize: 1 }
    })
    if (productsRes.result?.data?.list?.length > 0) {
      productId = productsRes.result.data.list[0].productId
      this._testProductId = productId
    }
  }
  
  if (!productId) {
    // 仍然没有，跳过测试
    return
  }
  
  // 使用商品ID创建订单
  const res = await wx.cloud.callFunction({
    name: 'orders',
    data: {
      action: 'create',
      items: [{ productId, quantity: 1 }]
    }
  })
}
```

**经验总结**：
测试用例之间可能有依赖关系。需要确保前置条件满足，或在测试中主动检查并获取必要的数据。同时，详细的日志输出对排查问题非常有帮助。

- 调用 `wx.cloud.callFunction` 测试云函数
- 没有传统意义上的代码覆盖率
- 需要用"功能覆盖率"来衡量

**解决方案**：
- 设计了 29 个测试用例覆盖核心功能

- 用"功能覆盖率"代替"代码覆盖率"

**收获**：不同的技术栈有不同的测试方法，需要因地制宜，找到适合自己项目的测试方式。
