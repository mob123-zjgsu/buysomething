===========================================
作业一：云服务部署 — 遇到的问题和解决
===========================================

【成员 A 使用 1-4 条，成员 B 使用 5-8 条】


--- 遇到的问题和解决 ---

1. 问题：静态网站托管开通后访问显示 404，没有任何页面
   解决：检查发现静态托管中没有任何文件（连 index.html 都没有）。通过 CloudBase MCP 工具将项目首页 HTML 文件上传到静态托管，并设置 index.html 为默认首页文档，刷新后页面正常显示。

2. 问题：GitHub Actions 工作流（cloudbase-deploy.yml）已编写但从未成功触发自动部署
   解决：排查发现 GitHub 仓库的 Secrets（TCB_SECRET_ID、TCB_SECRET_KEY）未配置，工作流在运行时因为没有 API 密钥而失败。在 GitHub Settings → Secrets and variables → Actions 中添加了对应的密钥值后，工作流可以正常连接 CloudBase 并部署云函数。

3. 问题：Vercel 部署时使用 ESM 语法（import/export）的 Express 应用频繁崩溃，返回 500 INTERNAL_SERVER_ERROR
   解决：经过多次调试发现，Vercel 的 serverless 函数环境对 ES Module 的跨文件导入支持不稳定，且 package.json 中 "type":"module" 的设置与 Vercel 构建器存在兼容问题。最终放弃 Express/CORS 依赖，改用纯 Node.js 原生 handler（CommonJS 格式 .cjs 文件），零第三方依赖，函数稳定运行。

4. 问题：Vercel 部署配置多次调整才成功，先后遇到了 404 NOT_FOUND 和 500 错误
   解决：最初使用 rewrites 路由模式时，Vercel 无法识别 .cjs 文件作为函数入口，报 404。改用 builds + routes 模式后路由正确但函数仍崩溃，最终排查是第三方依赖（express/cors）在 serverless 环境加载失败。总结出 Vercel 最稳定的部署模式：纯 Node.js handler + CommonJS + builds/routes 配置。

5. 问题：云函数创建时出现 "entryFile did not find in code or layers" 错误
   解决：发现是因为误把 cloudfunctions 目录名当成了函数名进行部署，cloudfunctions 是存放所有云函数的父目录而非单个函数，其中没有 index.js 入口文件。正确的做法是部署 cloudfunctions 下的各个具体函数子目录（如 health、login 等）。

6. 问题：微信小程序的前端页面（WXML/WXSS）无法部署到 Vercel 等网页托管平台
   解决：理解了微信小程序的运行机制——WXML/WXSS 只能在微信客户端内渲染，无法在浏览器中展示。因此将部署目标调整为"API 演示站 + 后端云函数"而非完整的前端 UI。在 Vercel 上部署了包含健康检查、商品 API 接口、状态展示页的 Node.js 服务。

7. 问题：多平台部署时配置文件分散在不同位置，管理混乱
   解决：梳理了项目的部署架构：CloudBase 负责后端云函数（cloudbase-deploy.yml）、Vercel 负责前端 API 服务（vercel.json）、Docker 负责容器化（Dockerfile）。统一将部署配置放在对应服务的根目录，文档中绘制了部署架构图说明各平台之间的关系。

8. 问题：环境变量在不同平台上的配置方式不同，容易遗漏
   解决：CloudBase 的环境变量在云函数控制台逐个配置，Vercel 的环境变量在项目 Settings → Environment Variables 中配置，GitHub Actions 的环境变量通过 Secrets 管理。整理了一份《环境变量配置对照表》记录每个平台需要哪些变量及配置路径，避免遗漏。


--- 心得体会 ---

1. 从本地到云端：通过这次作业完整经历了"本地开发 → 配置文件编写 → 平台注册 → 连接仓库 → 一键部署 → 公网访问"的全流程，真正理解了云服务部署的本质是把代码从自己电脑搬到服务商的服务器上运行。

2. 平台选择的策略：不同部署平台有不同定位——Vercel 擅长前端/全栈的快速部署（自动 HTTPS、Git 推送即部署），CloudBase 擅长微信生态的 Serverless 后端（云函数+数据库+存储一体），选对平台能让部署事半功倍。

3. 配置文件是部署的灵魂：vercel.json、Dockerfile、.github/workflows/*.yml 这些配置文件的本质是"用代码描述部署环境"，确保任何人在任何时间都能复现完全相同的部署结果，避免了"我这能跑你那不能跑"的问题。

4. CI/CD 自动化的价值：虽然手动部署也能完成任务，但 Git 推送自动触发的 CI/CD 流水线让每次代码变更都能自动上线，既减少了人工操作出错的可能性，也让团队协作时每个人都能看到最新的线上效果。

5. 调试 serverless 部署的教训：Vercel 的三次失败（500 → 404 → 500 → 成功）让我深刻体会到 serverless 环境与传统服务器环境的差异——不是所有代码都能原样搬上去跑，需要根据平台的运行时约束做适配。

6. 敏感信息管理是底线：API 密钥、数据库密码等敏感信息绝对不能硬编码在代码中，必须通过环境变量或 Secrets 管理。GitHub 上的一个不当提交就可能泄露生产环境凭据，这种安全意识是部署课程最重要的收获之一。

7. 文档和截图的必要性：部署完成后，清晰的文档和截图不仅是作业要求，更是日后维护和交接的依据。线上地址、配置位置、环境变量清单、故障排除步骤，这些记录在项目出问题时能大大缩短排查时间。

8. 静态托管的误区：最初以为"开通了静态托管 = 网站上线"，实际还需要手动上传文件并配置首页文档。云平台的各项服务之间是解耦的，开通服务只是创建了资源容器，往里放内容才是真正的"部署"。

9. 零依赖 serverless 函数的优势：最终成功的 Vercel 部署使用的是纯 Node.js handler（无 express、无 cors），这让我意识到在 serverless 场景下"少即是多"——每减少一个依赖就减少一个可能的故障点，冷启动也更快。

10. 部署即输出：部署不是一个"技术步骤"，而是"交付方式"。代码写完了不部署，等于没写——因为用户访问不到。让学生作业真正上线、能用 URL 打开，这种成就感远超交一份本地代码。
